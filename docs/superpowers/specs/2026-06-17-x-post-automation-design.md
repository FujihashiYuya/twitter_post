## 1. 目的とゴール

現状、投稿文の「作成（テーマ決め→裏取り→ルール参照→下書き）」までは Claude が支援できている。
本プロジェクトでは、ここに次の3つを追加して運用を半自動化する。

1. **自動投稿**: X API を使い、承認済みの投稿を設定時刻に自動で投稿する
2. **週次分析**: 各投稿のいいね/リポスト/返信/引用/インプレッション等を週1回取得・分析し、改善提案を出す
3. **スキル化**: 一連の「投稿作成ワークフロー」を Claude Code スキルとして整備する（既存ベストプラクティスを調査の上）

### 非ゴール（YAGNI / 今回はやらない）

- 自動いいね・自動リプライ・自動フォロー等のエンゲージメント自動化
- 画像生成・添付の自動化
- 複数アカウント運用
- A/Bテストの自動実行
- 投稿文を「投稿時刻にその場でLLMが創作する」こと（誤投稿防止のため明確に範囲外）

## 2. 確定した方針（意思決定ログ）

| 論点 | 決定 | 理由 |
|------|------|------|
| 投稿の承認フロー | **週まとめて事前承認** | アカウントのNGルールが厳格。人の承認を必須ゲートにする |
| 実行基盤 | **GitHub Actions（cron）**（当初Claude routinesから変更） | 決定論的な投稿/指標取得はGHAの方が確実・無料（cwd/ブランチpush/実行上限の制約が無い）。クラウド実行でPCに依存しない |
| ストレージ/受け渡し | **GitHub プライベートリポジトリ** | クラウドの routine はローカルPCを参照できないため、クラウド到達可能な置き場所が必要 |
| 設計原則 | **決定論=スクリプト / 思考=Claude** | 投稿・指標取得は Python に固定し誤投稿を防ぐ。週次分析は Claude の思考力を活用 |
| 週次分析の出力 | **数値台帳(CSV)への蓄積 ＋ 週次考察レポート(md)＋改善提案** | 時系列の定量蓄積と、目標照合の定性考察の両方を得る |
| 投稿時間 | **固定スロットにせず、曜日・時刻を意図的にばらして実験** | 反応の良い時間帯を統計的に探る。週次分析で時間帯×曜日別に集計 |

> **実装メモ（2026-06-22）**: スケジュール実行は **GitHub Actions** で実装・本番検証済み（`.github/workflows/post.yml`＝投稿、`metrics.yml`＝週次指標）。週次分析レポートの執筆のみ、オンデマンドのローカルClaudeスキル `/analyze-x-week` で実施。以降の §4.4/§4.5 に出てくる「routine A/B」は GitHub Actions の post.yml/metrics.yml に読み替えること。承認フロー・スクリプト・データ設計は不変。

### X API 料金の前提（2026-06 時点・確認済み）

- 旧無料枠は新規廃止。新規は**従量課金（月額固定なし）**が標準。
- 単価: 投稿作成 **$0.015/件**（**URLを含む投稿は$0.20/件**）、投稿読取 $0.005/件、**自分のデータの読取 $0.001/件**、ユーザー情報 $0.01/件。読取上限 200万/月。
- 旧 Basic($200/月)・Pro($5,000/月) は新規申込不可（既存契約者のみ）。
- 本用途の想定: 投稿 約16本/月（$0.015×16≈$0.24、リンク付きは$0.20/件で増）＋ 週次の自分データ読取 約64件/月（$0.001×64≈$0.06）→ **月 $0.3〜（≈50〜200円程度。リンク投稿が多いほど増える）**。
- 出典: X API pay-per-use announcement / X API Metrics docs / medianama(2026-02)。

        ▼                                        │                    ▼

## 4. コンポーネント設計（ユニットと責務）

各ユニットは「何をするか／どう使うか／依存」を1つに保つ。

### 4.1 `xtools/`（決定論的処理・Python）

リポジトリ直下に Python パッケージ `xtools/` を置く。投稿・取得の実処理はすべてここに固定する。

| モジュール | 何をするか | 入力 | 出力 | 依存 |
|-----------|-----------|------|------|------|
| `xtools/auth.py` | OAuth 1.0a ユーザー認証の署名セッションを生成（長期トークン・更新不要） | 環境変数（api_key/api_secret/access_token/access_token_secret） | 署名済みHTTPセッション | `requests-oauthlib` |
| `xtools/post_tweet.py` | 承認済み&投稿時刻到来の md を投稿し、結果を md に書き戻す | `post/*.md`、現在時刻 | 投稿（API）、md更新、ログ | `auth.py`、`postfile.py`、X tweets endpoint |
| `xtools/fetch_metrics.py` | 投稿済み md の tweet_id 群の指標を取得し台帳に追記 | `post/*.md`（投稿済み） | `analysis/metrics_log.csv` 追記 | `auth.py`、`postfile.py`、X tweets endpoint |
| `xtools/postfile.py` | md の frontmatter とスレッド本文をパース/更新する共通ロジック | md ファイルパス | パース結果 / 書き戻し | 標準ライブラリ + `pyyaml` |

**設計意図**: routine（Claude）は「`python -m xtools.post_tweet --due-now` を実行する」だけ。投稿内容の判断・整形は Claude にさせず、スクリプトが md の内容をそのまま投稿する。

### 4.2 投稿ファイル（第1層・データ）

既存の `post/*.md` を拡張する。**YAML frontmatter を先頭に追加**し、機械可読フィールドを持たせる。
frontmatter を持たない既存ファイルは「投稿対象外」として無視されるため後方互換。

```markdown
---
status: 下書き              # 下書き / 承認済み / 投稿済み / 却下
scheduled_at: 2026-06-20T07:00:00+09:00   # JSTオフセット付きISO8601。承認時に設定
category: 技術ログ           # 技術ログ / 学習習慣 / 個人開発 / ツール活用
thread: false               # スレッド投稿か
posted_at:                  # 投稿後に自動記入
tweet_ids: []               # 投稿後に自動記入（スレッドは複数）
---

# 投稿文

[1ツイート目の本文]

===

[2ツイート目の本文（スレッドの場合）]

---

## メタ情報
（従来の人間向けメタ情報。残置可）
```

- **スレッド区切り**: 本文中で行頭が `===`（イコール3つ以上のみの行）をツイート境界とする。`thread: false` のときは区切りを無視し本文全体を1ツイート扱い。
- **文字数**: 投稿前に各ツイートの重み付き長（CJK=2、その他=1）が 280 以下かを検証。超過時は投稿せず失敗ログ（CLAUDE.md の「140文字以内」方針と整合）。

### 4.3 承認フロー（第1層・運用）

1. Claude が下書きを `status: 下書き` で生成（第3層スキルが担当）。
2. 人が週1回まとめてレビュー。投稿するものを `status: 承認済み` にし、`scheduled_at` を投稿時刻に設定。却下は `status: 却下`。
3. `git commit && git push` した時点で「承認」が確定（クラウドの routine が読めるようになる）。

`scheduled_at` の時刻は、routine A の起動スロット（後述）のいずれかに合わせる。

### 4.4 routine A — 毎日の投稿（第2層）

- **スケジュール**: cron `0 9,12,15,18,20,22 * * *` / timezone `Asia/Tokyo`
  - 起動時刻＝「投稿候補時刻」。昼・午後・夕方・夜をカバーし、曜日と時刻をばらして投稿時間を実験できる幅を持たせる（1日6回。routinesの1日あたり実行上限を圧迫しない範囲）。
  - より細かく試したい場合は毎時起動（`0 * * * *`）に切替可（上限が許せば）。
- **プロンプト（要旨）**: 「リポジトリの `python -m xtools.post_tweet --due-now` を実行し、結果（投稿/スキップ/失敗）を要約せよ。md の変更を commit & push せよ。」
- **「投稿対象（due）」の判定**: `status == 承認済み` かつ `scheduled_at <= 現在時刻(JST)` かつ 未投稿（`tweet_ids` 空）。
  - `scheduled_at` の時刻は投稿候補時刻のいずれか（分は :00）に合わせる。
- **自己修復**: 過去分（取りこぼし）も「due」に含むため、ある起動を取りこぼしても次の起動で投稿される（routines に取りこぼし再実行はないため、この方式で吸収）。
- **投稿後処理**: `status: 投稿済み`、`posted_at`、`tweet_ids` を書き戻し、commit & push。

#### 4.4.1 投稿時間の実験（ローテーション）方針

「いつ投稿すると反応が良いか」を統計的に探るため、投稿作成・承認時に `scheduled_at` を**意図的に分散**させる。

- 候補時刻（JST）: **9 / 12 / 15 / 18 / 20 / 22**。曜日も平日・週末をばらす。
- 数週間かけて「曜日帯 × 時刻」の各セルにサンプルが貯まるよう割り当てる（例: 同じ時刻に偏らせない）。
- 週次分析(routine B)が `posted_at` から「曜日」「時刻」を導出し、時間帯別の平均インプレッション/エンゲージメント率を集計 → 反応の良い時間帯の仮説を毎週更新する。
- 第3層スキルが下書き生成時に、この分散方針に沿って `scheduled_at` 候補を提案する。

### 4.5 routine B — 週次分析（第2層）

- **スケジュール**: cron `0 12 * * 0`（毎週日曜 12:00）/ timezone `Asia/Tokyo`
- **処理**:
  1. `python -m xtools.fetch_metrics` を実行 → 投稿済み md の指標を取得し `analysis/metrics_log.csv` に追記。
     - 取得項目: impressions(インプレッション)、likes、retweets、replies、quotes、bookmarks（取得可なら）、url_link_clicks、user_profile_clicks。
     - `posted_at` から派生する `posted_weekday`（曜日）・`posted_hour`（時刻）も列として記録（時間帯実験の集計用）。
     - non-public 指標（クリック等）は「自分の投稿・直近30日」のみ取得可。週次なので窓内で取得できる。impressions/likes 等は public_metrics でも取得可。
  2. Claude が `metrics_log.csv` と `account.md` の目標（3ヶ月でフォロワー500人、平均インプレッション1,000 等）を照合し考察。
  3. `analysis/YYYY-Www_report.md` を生成（下記テンプレ）。commit & push。

**週次レポートテンプレート（`analysis/`）**
```markdown
# 週次レポート YYYY-Www (M/D〜M/D)

## サマリ
- 今週の投稿数 / 平均インプレッション / 平均エンゲージメント率
- 先週比の増減

## 投稿別パフォーマンス（上位/下位）
| 投稿 | カテゴリ | 曜日 | 時刻 | imp | いいね | RT | 返信 | eng率 |

## 時間帯×曜日 別パフォーマンス（投稿時間の実験集計）
| 曜日 | 時刻 | サンプル数 | 平均imp | 平均eng率 |
（蓄積データから、反応の良い時間帯の仮説を毎週更新）

## 考察（何が伸びた/伸びなかったか・要因仮説）
## 来週の改善提案（具体的な投稿テーマ・文体・投稿時間帯の調整案）
```

### 4.6 第3層 — 投稿作成スキル

- **スコープ**: 「テーマ決め → 裏取り（必要に応じWebSearch）→ account.md / CLAUDE.md 参照 → 下書き生成 → レビューチェックリスト適用 → frontmatter付き `post/*.md` に `status: 下書き` で保存」までの**作成ワークフロー**。投稿・指標取得は含めない（スクリプト/routine の責務）。
- **作成手順**:
  1. **既存ベストプラクティス調査**: 公開されている X/SNS 投稿系・コンテンツ作成系の Claude スキル/エージェント例（公式 anthropics 系・コミュニティ）を調査し、構成の参考にする。
  2. `writing-skills` の作法に従い `SKILL.md`（frontmatter: name / description=起動トリガを明記）＋必要なら `references/` を作成。
  3. 既存の CLAUDE.md チェックリスト（方針整合・リスク・フォーマット）をスキル内のステップに組み込む。
- **スキル名（仮）**: `create-x-post`（レビュー時に確定）。

## 5. データフロー（投稿1本のライフサイクル）

## 6. エラーハンドリング / 信頼性

| 事象 | 方針 |
|------|------|
| 二重投稿の防止（最重要） | 投稿成功直後に `status:投稿済み` を書き戻し commit&push。次回以降は未投稿条件で除外。push 失敗時の再投稿リスク低減のため、投稿前に `analysis/posted_ledger.json` に tweet本文ハッシュを確認/記録するガードを設ける |
| 投稿失敗（API/認証エラー） | 当該 md は `status:承認済み` のまま据え置き、ログに理由記録。次スロットで再試行 |
| 文字数超過 | 投稿せず失敗ログ。人が修正 |
| routine の取りこぼし（catch-up無し） | due 判定が「`scheduled_at <= now` かつ未投稿」のため次スロットで自動投稿（self-healing） |
| 読取上限/レート | 本用途は桁違いに小さく到達しないが、`fetch_metrics` は対象を投稿済みに限定し最小化 |
| non-public 指標の30日制限 | 週次取得で窓内に収める。窓を過ぎた分は public_metrics の範囲のみ記録 |
| OAuth トークン失効 | OAuth 1.0a の長期トークンは更新不要。401(失効/権限変更)時は投稿せず明示エラーで停止（誤投稿より停止を優先） |

## 7. セキュリティ / コンプライアンス

- **NGルール担保**: 投稿の最終ゲートは「人の承認」。スキルのレビューチェックリスト（会社名/守秘/政治宗教/断定・マウント表現 等）を通過したものだけを人が承認する。
- **秘密情報**: X の API Key / API Secret / Access Token / Access Token Secret（OAuth 1.0a）は routine 環境変数に格納。routine 環境変数は「その環境を編集できる人に見える」ため、**個人専用の環境**に分離。リポジトリは**プライベート**。トークンは絶対にコミットしない（`.gitignore` と `.env.example` を用意）。
- **ネットワーク許可**: routine 環境の許可ドメインに `api.x.com`（必要なら `api.twitter.com`）を追加。

## 8. 第0層 セットアップ手順（概要）

1. このフォルダで `git init` → `.gitignore` 追加 → GitHub プライベートリポジトリを作成し push。**本 spec のコミットもここで実施**。
2. X 開発者ポータルでアプリ作成。アプリ権限を **Read and write** に設定し、自分のアカウント用に **OAuth 1.0a の Access Token & Secret を発行**（リダイレクト不要・即発行）。従量課金の支払い方法を登録。
3. `xtools/` 一式と `requirements.txt`（`requests`, `requests-oauthlib`, `pyyaml` 等）を追加。
4. ローカルで `xtools` の単体動作をドライラン（`--dry-run`）で確認。
5. Claude routines を2本作成（A: 投稿 / B: 週次分析）。環境変数・許可ドメイン・cron・timezone を設定。
6. routine の「1回限り実行（one-off）」で疎通確認後、スケジュール有効化。
