## 4. 投稿の運用（方法B': cron-job.org → GitHub Actions 自動投稿）

> 2026-07-05 に方法A（X純正スケジューラ手動予約）から移行。方法Aを選んだ主因だった「GHA schedule の
> 大幅遅延（news-paper 実測: +2.3〜5.6時間）」は、外部スケジューラ cron-job.org → `workflow_dispatch`
> で解消（起動<1秒・分単位で正確）。最終チェックは日曜の承認時に行うため品質担保は変わらず、
> 手動予約のクリック作業と tweet_ids 書き戻しの週1待ちが消える。投稿時刻は目標±1〜2分（実験には十分）。

- **投稿方法**: 日曜に `status: 承認済み`＋`scheduled_at` を設定して push するだけ。cron-job.org が毎日
  9/12/15/18/20/22時 JST に `post.yml` を起動し、due な投稿を X API で自動投稿 → `投稿済み`＋`tweet_ids`＋`posted_at` を即時書き戻し。
- **週次バッチ（日曜20〜30分）**:
  1. 自動生成済みの週次レポート（`analysis/<YYYY>-W<WW>_report.md`、日曜11時にGHAが生成）を読む → 次週の出し方を微調整
  2. 翌週5枠のうち**4本**を `post/` の在庫から番号順に選び、本文を最終チェック（de-identify・文字数・トーン）
  3. 4本を `status: 承認済み`＋`scheduled_at`（**曜日・時刻をばらす**。候補: 9/12/15/18/20/22時 JST。Tue〜Thuが技術系に強い、夜と昼を散らす）にして push＝予約完了
  4. 返信ターゲットのリスト（`analysis/reply_targets.md`）を手入れ

### 外部スケジューラ設定（cron-job.org・一度だけ）
news-paper の README「セットアップ」と同一パターン。
1. **GitHub Fine-grained PAT**: Repository access = `twitter_post` のみ / Permissions = **Actions: Read and write**
2. **cron-job.org に2ジョブ**（Timezone = `Asia/Tokyo`、Method = `POST`、Body = `{"ref":"main"}`、
   Headers = `Authorization: Bearer <PAT>` / `Accept: application/vnd.github+json` / `X-GitHub-Api-Version: 2022-11-28`）:
   | ジョブ | スケジュール | URL |
   |---|---|---|
   | 投稿 | 毎日 9/12/15/18/20/22時 | `https://api.github.com/repos/<owner>/twitter_post/actions/workflows/post.yml/dispatches` |
   | 週次指標+分析 | 日曜 11:00 | `https://api.github.com/repos/<owner>/twitter_post/actions/workflows/metrics.yml/dispatches` |
3. **GitHub Secrets に `CLAUDE_CODE_OAUTH_TOKEN` を追加**（週次分析のClaude CLI用。news-paper と同じ値でよい）
4. ハマりどころ: URLは正確な `https://`・末尾スラッシュ無し（cron-job.org はリダイレクトを追わない）。dispatch は毎回本物の実行。

### ライブネタ枠（週1・在庫だけにしない）
- 週5枠のうち**1枠は予約せず残し、当日「その週に実際あったこと」を手動投稿**する（今日ハマったこと・いま作っているもの・学習の途中経過）。会話が生まれやすいのはライブネタで、返信エンジンとの相乗効果が目的。
- 当週にネタがなければ在庫から1本出してよい（フォールバック。無理に捻り出さない）。
- 投稿後は他と同じく md 化して `status: 投稿済み` で記録（重複防止のため）。

### スレッド枠（月1〜2本・保存される型）
- 月1〜2本は**5〜10ツイートのスレッド**を企画する（手順・チェックリスト・移行記など「保存される型」。1ツイート1論点）。
- 単発140字は流れて消えやすく、ブックマーク・プロフィール訪問を稼ぐのはスレッド。在庫の近縁テーマ（例: JWT/認可/OIDC、キャッシュ/TTL/レイテンシ）を束ねて再構成すると低コストで作れる。
- リンクを載せる場合は2ツイート目（=最初の返信）に置く（本文リンクはリーチ激減）。
- **コピー範囲**: X に貼るのは md の `# 投稿文` と `## メタ情報` の**間の本文だけ**（frontmatter・メタ情報は貼らない）。
- **投稿順**: `post/20260624_01〜26` は「脱クラスタ化した投稿順」。1本ずつは安全でも連投すると構成が推測されるため、AI系・認証系・データ系を週またぎで散らしてある。番号順に出せばよい。

### ステータス運用
- 通常ライフサイクル: `下書き` → （日曜レビューで）`承認済み`＋`scheduled_at` → （GHAが自動で）`投稿済み`＋`tweet_ids`＋`posted_at`。
- **`予約済み`** は「GHAを通さず手動 or X純正スケジューラで出す」**例外用**。自動投稿対象外で、
  週次の reconcile（`fetch_metrics.py`）がタイムライン突合で `投稿済み` に更新する（保険機構）。
  ライブネタ枠の手動投稿も、投稿後に md 化して `予約済み` にしておけば reconcile が tweet_id を拾う。
- 二重投稿防止: 同じ投稿を「承認済みにしてX純正予約もする」ことだけはしない（GHAとXの両方から出る）。

## 5. 立ち回りプレイブック（平日 15〜30分／最大の成長エンジン）

小規模アカウントの伸びの約半分は「自分の投稿」でなく「返信」から。平日は投稿せず（予約済み）、時間を返信に充てる。

```
① 3分      自分の前回投稿への反応に全返信（来た人と会話を続ける）
② 15〜20分  "少し上"の同分野アカウントへ価値ある返信 5〜8件 ← 最大のレバー
            ＋ 良い技術ポストを1〜2件 引用RT（自分の一言を添える）
③ 3〜5分   リスト/TLを読んで情報キャッチアップ（=次の投稿ネタ集め）／関連を数人フォロー
```

- **返信ターゲット**: `analysis/reply_targets.md` を毎日上から見て返信先を探す（アンカー＋5〜20倍規模の本命、発掘方法つき）。大物(数万)は埋もれるので狙わない。週次バッチで入れ替え。
- **効く返信**: データ・自分の経験・丁寧な問い・節度ある別視点を足す（著者が返信し返したくなる水準）。「すごい！」の一言はNG。
- **フォロー**: 1日5〜10人、関連のみ。フォロー/アンフォロー乱発はシャドウバン要因なのでNG。
- **やらない**: 相互フォロー、エンゲージメント・プール、「いいねしたら〜」系、煽り。短期で伸びても信頼を失い目的と逆行。
- **ハッシュタグ**: 原則なし（発見効果は薄く、3つ以上は逆効果）。**#駆け出しエンジニア/#エンジニアと繋がりたい は信頼を損なうので使わない**。健全な #個人開発 を本当に該当する投稿に1つだけは可。

## 6. KPI・ベンチマーク

- 主要KPI: **プロフィール訪問→フォロー転換率**（先行指標）／保存数／返信数。フォロワー数だけを追わない。
- 現実値: 無課金相当の通常運用で月+50〜150、1,000人まで6〜12ヶ月。Premium保有で上振れ余地あり。
